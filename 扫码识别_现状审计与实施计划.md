## 一、现状审计（对应文档第 18 节 1–5）

### 1.1 框架 / 添加页 / 数据结构 / 云函数 / 提醒

| 维度 | 现状 | 与文档差距 |
| --- | --- | --- |
| 框架 | 微信原生（WXML/WXSS/JS），无云开发 `wx.cloud` | 无服务端，所有「分层查询 / OCR / 订阅调度 / 幂等」只能先客户端实现或降级 |
| 添加页 `pages/add/*` | 已有「生产日期+保质期 / 直接到期日」二选一分段控件、保存前到期日+来源预览、分类网格后置、必填/更多信息折叠、键盘+安全区适配 | **无扫码入口、无商品模板/GTIN/code_type、无重复扫码合并提示、无 scan_session_id** |
| 库存批次数据结构 | 记录已含 `id,name,brand,category,productionDate,shelfLife,expiryDate,quantity,remaining_quantity,business_status,unit,price,storage,opened,openedDate,openedShelfLife,events[],expiry_source,createdAt` | 基本对齐文档 10.2 的「库存批次」；**缺 `product_id / gtin / code_type / lot_number / scan_session_id / effective_expiry_date / updated_at`** |
| 云函数 | 无 | 文档 12/15.2 的服务端能力全部未实现（属阶段 3 / 后台范围） |
| 提醒 | 站内提醒已有（总览优先级列表 + 提醒页按剩余天数过滤）；`wx.requestSubscribeMessage` 仅占位模板 `'your_template_id_1/2'`，无真实模板、无调度 | 站内提醒达标；微信订阅消息为桩，需真实模板 ID + 类目能力（正式环境验证，不可承诺永久） |

### 1.2 `wx.scanCode` 现有调用

- **结论：当前项目完全没有调用 `wx.scanCode`，无任何扫码入口、无任何 `scanType` 解析逻辑。**
- 即扫码功能从零开始。需新增扫码页，按文档 9.1 / 15.1 处理 `scanType`（EAN-13/8、UPC、QR、Data Matrix），**不得用字符串长度判断类型**。

### 1.3 商品信息来源 / 覆盖率 / 接口授权

- **结论：无任何商品信息来源、覆盖率 0%、无接口授权。**
- 无个人模板库、无自建商品库、无第三方 API。
- 文档 7.1 要求的分层查询（个人模板 → 本地缓存 → 自建库 → 第三方 → 手动）当前只能做到「未命中→极简手动录入」。
- 真实第三方覆盖率需阶段 0 验证 + 服务端，未通过前**不得对外宣称自动识别比例**（文档 19）。

### 1.4 商品模板 / 库存批次 / 库存事件 是否已分离

| 概念 | 现状 |
| --- | --- |
| 库存批次（batch） | ✅ 已实现（每条食品记录 = 一个批次，含 `remaining_quantity`/`business_status`） |
| 库存事件（event） | ✅ 已实现（`events[]` 记录 create/update/consumed/discarded/partial，含时间戳；已过期不自动生成浪费事件，符合文档 10.4） |
| 商品模板（product template） | ❌ **未分离**。商品身份（名称/品牌/分类/规格）直接内嵌在每条批次里，无 `product_id`、无模板存储。重复扫码无法复用历史资料（文档 6.2.6 / 7.1/7.2 无法落地） |

> 阶段 A 已搭好「商品+批次+事件」地基，唯独缺「商品模板」这一层。这是扫码功能的核心前置依赖。

### 1.5 日期算法 / 重复扫码 / 断网 / 幂等

- **日期算法 ✅ 一致**：`daysUntil` 用自然日 `Math.round`（双方对齐 0 点），`getStatus` 边界互斥（expired<0 / urgent 0~1 / soon 2~3 / ok 4~7 / safe>7）。总览/食品库/详情/提醒均调用 `utils/data.js` 同一套；仅 `app.js` 内重复了一份等价实现（建议改为复用，消除漂移风险）。
- **重复扫码 ❌**：尚未有扫码，无合并/新增批次逻辑。文档 6.6 要求：默认新增批次，仅当 `gtin+到期日+位置+单位` 全一致且用户确认才提示合并。
- **断网保存**：当前纯本地存储，无云端，故「断网虚假成功」风险暂不存在；若后续上云需加「待同步」状态（文档 14）。
- **幂等 ❌**：无 `scan_session_id`、无防抖去重。扫描连续触发/重复点击需客户端防抖 + 会话 ID（文档 14/15.3）。

## 二、阶段 0：能力验证方案（文档 16 / 18 第 6 点）

> 阶段 0 **只做技术验证与数据评估，不承诺上线范围**，不对外宣称覆盖率。

| 验证项 | 方法 | 通过标准 |
| --- | --- | --- |
| 扫码类型支持 | 真机调用 `wx.scanCode({onlyFromCamera:false})`，打印 `scanType`，覆盖 EAN-13/8、UPC、QR、Data Matrix | 4 类均能正确返回类型与内容 |
| 个人模板闭环 | 用本地 `product.js` 缓存：扫同一码第二次应命中上次确认的资料 | 二次扫描复用名称/品牌/分类，不重复录入 |
| 手动降级 | 扫一个不存在的码 / 相册模糊图 | 不报错，可极简手动录入并完成入库 |
| 订阅消息能力 | 申请真实模板 ID，在「添加首件后」触发 `requestSubscribeMessage` | 真机收到授权；发送需服务端（阶段 3） |
| OCR 可行性 | 评估端侧 vs 云函数 OCR（包装喷码/压印/多格式） | 给出阶段 2 选型建议（本期不做） |
| 第三方授权 | 选定候选商品库，确认商用授权/合法域名/额度 | 满足条件再接入（本期不接） |

**阶段 0 交付：** 真机扫码类型测试记录 + 本地模板闭环 demo + 订阅授权真机验证截图 + 阻塞项清单。

## 三、阶段 1：扫码录入 MVP — 文件级改造计划（文档 16 阶段 1）

目标：标准条码/二维码扫描 → 分层查询（个人模板优先）→ 命中/未命中/超时/拒权均降级手动 → 日期二选一且强制确认 → 模板与批次分离 → 重复扫码默认新增批次、满足条件才合并 → 站内到期状态与详情可用。

### 3.3 关键规则落地（严守文档红线）

- 普通商品码 **只识别身份**，绝不伪造生产/到期日；日期必须经用户确认（文档 6.1/8.1）。
- 查询失败/超时/限流 → 全部降级手动录入，不阻塞（文档 6.2/14）。
- 同一 `gtin` 默认**新增批次**；仅 `gtin+到期日+位置+单位+在管` 全一致才提示「新增批次 / 合并数量」，不静默叠加（文档 6.6）。
- 不自动打开外部网页二维码；自建二维码不含令牌/身份（文档 6.3/6.4/13）。
- 扫描成功 ≠ 入库成功：确认页保存后才建批次（文档 15.3）。
- 第三方密钥只在服务端；本期纯前端不接第三方密钥（文档 13）。

### 3.4 数据迁移

- `normalizeFood` 给旧记录补 `product_id/gtin/code_type/scan_session_id` 默认值，旧批次不受影响，无需用户操作。
- 存量食品无 `gtin`，扫码新批次与之共存无冲突（靠 `id` 区分）。

### 3.5 测试与验收

- `node --check` 全量语法校验；对 `parseScanResult` / 模板查询 / 重复批次合并判定 / 日期算法 写纯函数单测。
- 真机走查文档 17 清单（扫描类型、降级、日期确认、合并、权限拒绝、暗光反馈、键盘不遮挡）。
- 上线前：`__usePrivacyCheck__` 改回 `true` 并在隐私协议声明相机/相册用途；清空预置示例数据。
